# MCP Agent Gateway 设计决策记录 (ADR)

## ADR-001: 选择 JSON-RPC 2.0 作为核心协议

### 状态
已采纳

### 背景
需要选择一种协议来标准化 MCP 网关与各 AI 平台之间的通信。

### 决策
选择 JSON-RPC 2.0，因为：
1. MCP 协议（Model Context Protocol）官方规范基于 JSON-RPC 2.0
2. 请求/响应模型简单可靠，易于调试
3. 支持 HTTP、SSE、STDIO 多种传输层
4. 与 MCP 生态兼容（Claude Desktop、Cursor、Trae 等）

### 替代方案
- **gRPC**: 性能更好，但浏览器兼容性差，MCP 生态不支持
- **GraphQL**: 灵活但过于复杂，MCP 不需要
- **自定义协议**: 不兼容 MCP 生态

## ADR-002: 自研 Agent 管线 vs LangGraph

### 状态
已采纳（自研管线，预留 LangGraph 扩展接口）

### 背景
项目需要 Agent 能力来编排多工具调用，LangGraph 是 LangChain 生态的成熟方案。

### 决策
选择自研简单的三段式管线（Planner → Executor → Validator），原因：
1. **场景匹配**: 本项目的 Agent 主要是工具调用编排，不需要 LangGraph 的循环、分支、子图等高级特性
2. **零依赖**: 不引入 LangChain 全家桶，降低部署复杂度（~200MB → ~50KB）
3. **可理解性**: 三段式管线比图状态机更直观，便于调试和维护
4. **渐进式**: 预留了 `IToolRegistry` 接口，未来可无缝接入 LangGraph

### 替代方案
- **LangGraph**: 功能强大但过重，对当前场景过度设计
- **CrewAI**: 多 Agent 框架，但当前只需要单 Agent 工具调用
- **AutoGen**: 微软多 Agent 框架，同样过于复杂

## ADR-003: 自研 LRU 缓存 vs LangChain Memory

### 状态
已采纳（自研 IncrementalContextCache）

### 背景
Agent 调用工具时，多次调用同一工具（如读取同一文件）会产生大量重复 Token 消耗。

### 决策
选择自研基于 LRU + SHA256 内容哈希的缓存，原因：
1. **缓存对象不同**: LangChain Memory 缓存对话消息，我们需要缓存工具调用结果
2. **内容感知**: 相同内容（如相同文件内容）只存一份，用哈希引用
3. **增量压缩**: 超长结果自动压缩，减少 Token 消耗
4. **零依赖**: 纯标准库实现（OrderedDict + hashlib）

### 替代方案
- **LangChain Memory**: 只缓存对话，不缓存工具结果
- **Redis**: 增加外部依赖，本地开发场景不友好
- **functools.lru_cache**: 不支持内容去重和增量压缩

## ADR-004: 选择 aiohttp 作为 HTTP 框架

### 状态
已采纳

### 背景
网关需要 HTTP 服务器来提供 MCP 和 REST API 服务。

### 决策
选择 aiohttp，原因：
1. **底层控制**: 对 SSE 和 Streamable HTTP 的原生支持更好
2. **轻量**: 比 FastAPI 依赖更少
3. **异步原生**: 原生 asyncio 支持，与工具执行匹配
4. **WebSocket 支持**: 未来扩展需要

### 替代方案
- **FastAPI**: 更流行但依赖更重，SSE 需要额外处理
- **Flask**: 同步框架，不适合异步工具调用
- **Sanic**: 类似 aiohttp 但生态较小

## ADR-005: 多租户隔离方案

### 状态
已采纳

### 背景
同一网关需要服务多个团队/项目，需要隔离不同租户的资源和权限。

### 决策
选择基于 API Key 分组的多租户方案：
1. 每个 API Key 属于一个租户（Tenant）
2. 每个租户有独立的文件白名单和工具权限
3. 请求通过 API Key 自动识别租户
4. 配置驱动，无需数据库

### 替代方案
- **数据库驱动**: 灵活性更高但增加运维复杂度
- **JWT Token**: 更标准但需要密钥管理
- **OAuth 2.0**: 过于复杂，内部使用场景不需要

## ADR-006: 审计日志使用内存环形缓冲区

### 状态
已采纳

### 背景
需要记录所有工具调用以便审计和问题排查。

### 决策
选择内存环形缓冲区（`collections.deque`），原因：
1. **零依赖**: 不引入外部存储
2. **固定内存**: 最多保留 1000 条记录，不会 OOM
3. **高性能**: 写入 O(1)，不阻塞工具调用
4. **可扩展**: 预留了持久化接口

### 替代方案
- **SQLite**: 持久化但写入性能较低
- **Kafka**: 适用于大规模场景，但增加运维复杂度
- **文件日志**: 简单但查询不便

### 后续改进
- 当流量增大时，可切换为 SQLite WAL 模式持久化
- 可添加 Kafka 适配器用于大规模场景

## ADR-007: 并行调度基于依赖图拓扑排序

### 状态
已采纳

### 背景
当 Agent 同时调用多个独立工具时，串行执行浪费总时间。

### 决策
选择基于依赖图拓扑排序的并行调度：
1. 分析工具之间的依赖关系（如 read_file → parse_config）
2. 构建有向无环图（DAG）
3. 按拓扑层级分组，同层工具并行执行
4. 总耗时 = 各层耗时最大值之和

### 替代方案
- **asyncio.gather()**: 简单粗暴但不支持依赖关系
- **Celery**: 分布式任务队列，对于单机场景过重
- **Ray**: 适合大规模并行，但引入重量依赖

## ADR-008: 配置使用 YAML 文件

### 状态
已采纳

### 背景
网关需要配置安全策略、租户、传输模式等。

### 决策
选择 YAML 配置文件，原因：
1. **可读性**: 比 JSON 更适合人工编辑
2. **注释支持**: YAML 支持注释，JSON 不支持
3. **层次结构**: 天然支持嵌套配置
4. **Python 标准库**: PyYAML 是事实标准

### 替代方案
- **JSON**: 不支持注释，不便于维护
- **TOML**: 适合简单配置，嵌套复杂时不直观
- **环境变量**: 不适合结构化配置

---

## ADR-009: 工具提供者使用抽象基类

### 状态
已采纳

### 背景
需要支持多种工具提供者（文件系统、终端、数据库、LLM、Web），且允许用户自定义扩展。

### 决策
使用 `BaseToolProvider` 抽象基类：
1. 定义统一的 `get_tools()` 和 `call_tool()` 接口
2. 通过 `ToolRegistry` 注册和查找
3. 支持前缀命名（如 `fs_read_file`、`db_query`）
4. 新增工具提供者只需继承基类并实现两个方法

### 替代方案
- **函数注册**: 更简单但缺少类型约束
- **插件系统**: 更灵活但过度设计
- **配置驱动**: 不够灵活，无法实现复杂工具逻辑
