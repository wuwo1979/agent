## 二、优化 1：增量上下文缓存

### 2.1 问题分析

在 Agent 多轮工具调用中，大量工具结果被重复传回 LLM：
- 相同参数调用相同工具 → 结果完全相同
- 大文件多次读取 → 每次都传回完整内容
- 数据库查询 → 相同 SQL 重复执行

### 2.2 解决方案

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ 工具调用请求   │ ──► │ 内容哈希去重   │ ──► │ 返回缓存结果   │
└──────────────┘     └──────────────┘     └──────────────┘
                            │
                     ┌──────┴──────┐
                     │ 命中？       │
                     └──────┬──────┘
                        N   │   Y
                     ┌──────┴──────┐
                     │ 执行工具     │
                     └──────┬──────┘
                            │
                     ┌──────┴──────┐
                     │ 压缩结果     │
                     └──────┬──────┘
                            │
                     ┌──────┴──────┐
                     │ 存入缓存     │
                     └──────────────┘
```

### 2.3 核心策略

| 策略 | 说明 | 效果 |
|------|------|------|
| 内容哈希去重 | 相同结果不重复存储/传回 | 减少重复传输 |
| 增量压缩 | 只传回变化部分（diff） | 减少增量传输 |
| 智能截断 | 超长结果自动截断 + 摘要 | 控制 Token 上限 |
| LRU 淘汰 | 限制缓存大小，淘汰冷数据 | 控制内存占用 |

### 2.4 实测数据

```
8 次工具调用，3 次缓存命中（37.5% 命中率）
全量回传 Token：~5628
缓存后 Token：~2844
Token 压缩率：49.5%
```

### 2.5 关键代码

```python
class IncrementalContextCache:
    def get(self, tool_name, arguments):
        key = self._compute_key(tool_name, arguments)
        if key in self._cache:
            self._cache.move_to_end(key)  # LRU
            return self._cache[key].content
        return None
    
    def set(self, tool_name, arguments, content):
        key = self._compute_key(tool_name, arguments)
        content_hash = self._compute_content_hash(content)
        
        # 内容去重
        if content_hash in self._content_hashes:
            return 0  # 相同内容，不重复存储
        
        # 压缩 + 存储
        compressed = self._compress(content)
        self._cache[key] = CacheEntry(content=compressed)
```

## 三、优化 2：无依赖工具并行调度

### 3.1 问题分析

传统 Agent 串行执行工具调用：
```
Tool A (100ms) → Tool B (100ms) → Tool C (100ms) = 300ms
```

无依赖工具可以并行执行：
```
Tool A ─┐
Tool B ─┤ → 100ms（取最慢的）
Tool C ─┘
```

### 3.2 依赖图分析

```
有依赖关系：
  A (无依赖) ─┐
  B (无依赖) ─┤ → 并行执行
  C (依赖 A, B) → 等待 A, B 完成
  D (依赖 C) → 等待 C 完成
```

```python
dependencies = {

    "B": [],           # 无依赖
    "C": ["A", "B"],   # 依赖 A 和 B
    "D": ["C"],        # 依赖 C
}
```

### 3.4 实测数据

## 四、优化 3：上下文压缩器

### 4.1 问题

工具返回的大 JSON 结果包含大量冗余信息：

```json
// 原始结果（14369 字符）
{
  "results": [
    {"id": 0, "name": "item_0", "data": "xxxxx..."},
    {"id": 1, "name": "item_1", "data": "xxxxx..."},
    // ... 100 条记录
  ]
}
```

### 4.2 压缩策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 批量截断 | 只保留前 N 条 + 总数 | 列表结果 |
| 深度限制 | 展平嵌套，限制深度 3 | 嵌套 JSON |
| 字段精简 | 移除冗余字段 | 固定结构 |
| 增量差异 | 只返回变化部分 | 重复查询 |

### 4.3 实测数据

```
原始大小：14369 字符（~3592 tokens）
压缩后：130 字符（~32 tokens）
压缩率：99.1%
```
