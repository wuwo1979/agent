## 二、高频面试题

### 2.1 MCP 协议

**Q1: 什么是 MCP 协议？它解决了什么问题？2026 年生态如何？**

A: MCP（Model Context Protocol）是 Anthropic 于 2024 年 11 月推出、2025 年 12 月捐赠 Linux 基金会的开放协议，用于标准化 AI 应用与外部工具/数据源之间的通信。

截至 2026 Q2：10,000+ 活跃 Server、9,700 万月 SDK 下载量、ChatGPT/Claude/Gemini 全面支持。

它解决了三个核心问题：
1. **工具调用标准化**：不同 AI 应用使用统一 JSON-RPC 2.0 协议调用工具，避免了 N×M 适配问题
2. **安全上下文隔离**：通过 Host-Client-Server 三层架构，将工具执行与 LLM 推理分离
3. **可扩展性**：插件化 ToolProvider 模式，新工具只需实现 `call_tool` 接口即可接入

2026 关键演进：
- 渐进式发现（Progressive Discovery）：按需加载工具
- 程序化工具调用（Programmatic Tool Calling）：Agent 生成代码批量执行
- 无状态传输（Stateless Transport）：Google 参与推进，支持 Serverless 部署
- MCP + A2A + AG-UI 三大协议分工：分别负责工具调用、Agent 协作、用户交互

**实现要点**：
- 传输层：支持 Streamable HTTP、SSE、STDIO 三种传输方式
- 协议层：完整的 JSON-RPC 2.0 实现，包括 `initialize`、`tools/list`、`tools/call` 等标准方法
- 安全层：API Key 认证 + 令牌桶速率限制 + 危险工具权限策略

**Q2: 你如何实现自定义 MCP 工具？**

A: 通过继承 `BaseToolProvider` 基类，实现两个核心方法：
```python
class MyToolProvider(BaseToolProvider):
    def _register_tools(self):
        self._register_tool(ToolDefinition(
            name="my_tool",
            description="My custom tool",
            inputSchema={"type": "object", "properties": {...}},
            category="custom",
        ))

    async def call_tool(self, tool_name, arguments):
        # 实现具体的工具逻辑
        result = do_something(arguments)
        return ToolCallResult.text_result(tool_name, result)
```

**Q3: MCP 协议中的 initialize 握手流程是怎样的？**

A: 客户端发送 `initialize` 请求（携带 `protocolVersion` 和 `clientInfo`），服务端返回 `serverInfo` 和 `capabilities`（包括 `tools` 能力声明）。握手成功后，客户端调用 `tools/list` 获取可用工具列表，随后即可正常调用 `tools/call`。

### 2.2 A2A 协议（2026 新增）

**Q3.5: 什么是 A2A 协议？和 MCP 的关系是什么？**

A: A2A（Agent-to-Agent）是 Google 于 2025 年 4 月推出、2026 年 3 月发布 v1.0 的开放协议，用于标准化 Agent 之间的通信。

| 维度 | MCP | A2A |
|------|-----|-----|
| 通信对象 | Agent → Tool | Agent → Agent |
| 状态 | 无状态（单次调用） | 有状态（Task 生命周期） |
| 能力发现 | `tools/list` | Agent Card（`/.well-known/agent.json`） |
| 任务管理 | 无 | Task 创建/查询/取消/订阅 |
| 认证 | OAuth 2.0 | OAuth 2.0 + mTLS + DID/VC |

二者互补不冲突：MCP 负责 Agent 调用工具，A2A 负责 Agent 之间协作。生产中常一起使用。

**Q3.6: 你的项目如何体现 A2A 思想？**

A: 项目的 Supervisor-Worker 模式本质上实现了 A2A 的核心概念：
- Supervisor 相当于 A2A Client Agent（任务发起方）
- Workers 相当于 A2A Server Agent（任务执行方）
- `_delegate_to_*` 相当于 A2A 的 `message/send`
- `run_streaming()` 相当于 A2A 的 `message/stream`
- AgentState + SnapshotManager 相当于 A2A 的 Task 状态管理
- 架构完全兼容 A2A 协议规范，可直接迁移到 A2A SDK

### 2.3 LangGraph & Agent 调度

**Q4: 你项目中 LangGraph 的状态图是如何设计的？**

A: 使用三节点循环架构：
- **Planning 节点**：LLM 将用户输入拆解为子任务 DAG（依赖有向无环图）
- **Execution 节点**：按拓扑排序分层并行执行无依赖工具
- **Validation 节点**：检查执行结果质量，不合格则返回 Planning 重新规划

状态通过 `AgentState` 管理，包含任务状态、执行计划、工具结果、错误信息等，通过 `Checkpointer` 持久化支持断点续跑。

**Q5: 什么是 Supervisor-Worker 模式？为什么选择它？**

A: Supervisor-Worker 模式是 Multi-Agent 架构的核心模式：
- **Supervisor**：中心协调者，负责动态路由、Handoffs（交接）、流式输出
- **Workers**：Planner（任务拆分）、Executor（工具执行）、Validator（结果校验）

优势：
1. 职责分离：每个 Worker 专注单一职责，易于调试和优化
2. 动态路由：Supervisor 根据执行状态动态决定下一步
3. 可观测性：每个 Worker 独立采集指标，便于性能分析
4. 容错性：单个 Worker 失败不影响 Supervisor 继续调度

**Q6: 如何实现工具调用的失败重试？**

A: 实现三层容错机制：
1. **指数退避重试**：失败后等待 1s、2s、4s 重试，最多 3 次
2. **熔断器**：连续失败 5 次后熔断 30 秒，防止雪崩
3. **降级策略**：熔断后执行降级函数（如返回缓存结果）

```python
class RetryManager:
    async def execute_with_retry(self, func, *args, fallback=None, **kwargs):
        if not self.circuit_breaker.allow_request():
            return await self._call_fallback(fallback, *args, **kwargs)
        for attempt in range(self.max_retries):
            try:
                return await func(*args, **kwargs)
            except Exception as e:
                if attempt == self.max_retries - 1:
                    raise
                await asyncio.sleep(2 ** attempt)
```

### 2.3 性能优化

**Q7: 你的增量上下文缓存是如何实现的？**

A: 核心思路是通过内容哈希去重避免重复传输：

1. **缓存键生成**：`hash(tool_name + json.dumps(arguments))` 作为缓存键
2. **命中判断**：调用工具前先查缓存，命中则直接返回缓存结果（Token 节省 100%）
3. **LRU 淘汰**：缓存满时淘汰最久未使用的条目
4. **结果压缩**：大型 JSON 结果通过 `ContextCompressor` 压缩为摘要

实际效果：Token 压缩率 49.5%，缓存命中率 37.5%。

**Q8: 无依赖工具并行调度是如何实现的？**

A: 使用拓扑排序分层并行：
1. 构建依赖图（DAG）：节点=工具，边=依赖关系
2. 拓扑排序分组：无依赖的节点在同一层
3. 层内并行：同一层的工具用 `asyncio.gather` 并行执行
4. 层间串行：上层完成后才执行下层

实际效果：6 个工具并行执行，加速比 2.9x，耗时降低 65%。

**Q9: 多模型适配器如何设计？**

A: 统一接口 + 适配器模式：
```python

    @abstractmethod

    @abstractmethod
```

## 四、面试中可能遇到的追问

**Q: 你的系统和 LangChain 的 Agent 有什么区别？**

A: LangChain 的 AgentExecutor 是线性循环模式（Thought → Action → Observation → 循环），而我们使用了 Supervisor-Worker 模式，优势在于：
- 职责分离更清晰（规划/执行/校验独立）
- 支持并行工具调用（拓扑排序分层）
- 内置性能优化（缓存、压缩、适配器）
- 完整的可观测性（Worker 指标采集）

**Q: 如果工具调用一直失败怎么办？**

A: 三层保护：
1. 单次重试（指数退避，最多 3 次）
2. 熔断保护（连续失败 5 次熔断 30 秒）
3. 全局重试（整个 Agent 流程重新规划，最多 3 次）

**Q: 如何处理大文件读取导致的 Token 爆炸？**

A: 两层优化：
1. 增量缓存：相同文件读取只传一次，后续命中缓存
2. 上下文压缩：大型 JSON 结果压缩为摘要（实测压缩率 99.1%）

**Q: 你的项目有什么不足？**

A: 诚实地回答：
1. 目前是简化版 LLM Agent（无真实 LLM 时使用规则匹配），完整效果需要接入 LLM
2. 未实现多人协作和权限管理
3. 监控面板（Grafana）还在规划中
4. 可以进一步优化为支持分布式部署
