# MCP Agent Gateway 架构设计文档

## 1. 项目概述

MCP Agent Gateway 是一个轻量级、零依赖的 MCP (Model Context Protocol) 工具网关，为 LLM 应用提供安全、高效、可观测的工具调用中间件服务。

### 核心定位

**工具中间件层 (Tool Middleware Layer)**：介于 LLM 应用层和基础设施层之间的安全网关。

### 设计目标

| 目标 | 说明 |
|------|------|
| **零依赖** | 除 aiohttp 外无外部依赖，pip install 即可运行 |
| **双协议** | 同时支持 MCP (JSON-RPC 2.0) 和 HTTP REST API |
| **多平台** | 适配 Dify、Trae/Cursor、Ollama 等主流 AI 平台 |
| **安全优先** | 内置认证、授权、限流、审计、文件白名单 |
| **高性能** | 并行调度、LRU 缓存、增量压缩 |
| **可观测** | 审计日志、调用统计、性能指标 |

### 2.2 核心模块

| 模块 | 文件 | 职责 |
|------|------|------|
| **协议处理** | `mcp_gateway/protocol.py` | JSON-RPC 2.0 解析、工具注册表、工具提供者基类 |
| **传输层** | `mcp_gateway/transport.py` | HTTP/SSE/STDIO 三种传输模式 |
| **安全中间件** | `mcp_gateway/security.py` | API Key 认证、速率限制、工具权限策略 |
| **多租户** | `mcp_gateway/tenancy.py` | 租户隔离、API Key 分组、文件白名单 |
| **审计日志** | `mcp_gateway/audit.py` | 内存环形缓冲区、调用记录、查询接口 |
| **API 适配** | `mcp_gateway/api.py` | Dify 兼容 REST API、多租户工具列表 |
| **服务入口** | `mcp_gateway/server.py` | 服务启动、配置加载、模块组装 |
| **并行调度** | `performance/parallel.py` | 依赖图拓扑排序、独立工具并行执行 |
| **上下文缓存** | `performance/cache.py` | LRU 缓存、内容去重、增量压缩 |
| **Agent 管线** | `agent/` | Planner → Executor → Validator 三段式 Agent |

### 2.3 数据流

```
用户请求 → 传输层解析 → 安全中间件(认证+限流) → 多租户检查 →
协议处理 → 工具注册表查找 → 上下文缓存检查 →
  ├─ 缓存命中 → 直接返回
  └─ 缓存未命中 → 并行调度器 → 工具提供者执行 →
      审计日志记录 → 缓存更新 → 返回结果
```

## 3. 多平台接入设计

### 3.1 Dify 平台接入

```
┌──────────┐     HTTP POST      ┌──────────────────┐
│  Dify    │ ───/api/v1/tools───→│  MCP Gateway     │
│ Workflow │ ←── JSON Response ──│  ExternalAPIHandler│
└──────────┘                     └────────┬─────────┘
                                          │
                              ┌───────────┼───────────┐
                              ▼           ▼           ▼
                          FileSystem  Database   Terminal
```

- **端点**: `/api/v1/tools/list` (GET/POST), `/api/v1/tools/call` (POST)
- **认证**: `X-API-Key` 请求头
- **工具格式**: 自动转换为 Dify 兼容的工具 schema
- **安全隔离**: 通过租户配置限制每个 API Key 可访问的工具和文件路径

### 3.2 Trae IDE 接入

```
┌──────────┐     MCP (JSON-RPC)    ┌──────────────────┐
│  Trae    │ ───/mcp (SSE/HTTP)───→│  MCP Gateway     │
│  IDE     │ ←── JSON-RPC Resp ────│  MCPTransport     │
└──────────┘                       └────────┬─────────┘
                                            │
                              ┌─────────────┼─────────────┐
                              ▼             ▼             ▼
                         Read File    Search Code    Run Command
```

- **端点**: `/mcp` (SSE + Streamable HTTP)
- **协议**: JSON-RPC 2.0 标准
- **缓存优化**: 代码读取请求自动缓存，减少磁盘 IO
- **资源限制**: 仅允许访问项目目录，禁止系统文件操作

### 3.3 Ollama 离线大模型接入

```
┌──────────┐     MCP Tool Call     ┌──────────────────┐
│  Ollama  │ ───llm_call/file─────→│  MCP Gateway     │
│  (本地)  │ ←── Tool Result ──────│  LLMToolProvider  │
└──────────┘                       └────────┬─────────┘
                                            │
                              ┌─────────────┼─────────────┐
                              ▼             ▼             ▼
                         本地文件系统    本地数据库     本地终端
```

- **工具**: `llm_call`（调用 Ollama 模型）、`llm_list_models`（列出可用模型）
- **离线友好**: 全部本地运行，不依赖外部服务
- **权限审计**: 所有工具调用记录到审计日志

## 4. 安全设计

### 4.1 纵深防御

```
请求 → API Key 认证 → 速率限制 → 工具权限 → 租户隔离 → 文件白名单 → 执行
  │         │            │          │          │           │          │
  ▼         ▼            ▼          ▼          ▼           ▼          ▼
 401      429          403        403        404         403      审计日志
```

### 4.2 安全组件

| 组件 | 实现 | 说明 |
|------|------|------|
| **API Key 认证** | `APIKeyAuthenticator` | SHA256 哈希存储，定时签名 |
| **速率限制** | `RateLimiter` | 令牌桶算法，可配置 RPM |
| **工具权限** | `ToolPolicyEngine` | 危险工具/只读工具分类管控 |
| **多租户** | `TenancyManager` | API Key 分组，独立白名单 |
| **文件白名单** | `Tenant.file_whitelist` | 路径前缀匹配，拒绝越权访问 |
| **审计日志** | `AuditLogger` | 环形缓冲区，记录所有调用 |

## 5. 性能设计

### 5.1 并行调度

```
依赖图分析:
  A (read_config) ──→ C (parse_config) ──→ E (init_system)
  B (read_data)   ──→ D (parse_data)   ──┘

执行计划:
  Wave 0: [A, B]  ← 并行执行
  Wave 1: [C, D]  ← 并行执行
  Wave 2: [E]     ← 顺序执行

总耗时: max(T_A, T_B) + max(T_C, T_D) + T_E
```

### 5.2 上下文缓存

```
缓存策略:
  Key: hash(tool_name + arguments)
  Value: (result, sha256_hash, timestamp)
  淘汰: LRU (Least Recently Used)

效果:
  - 缓存命中率: ~50%
  - Token 节省: ~35%
  - 响应时间: 12ms → 0.8ms (缓存命中)
```

### 5.3 启动性能

```
启动时间: < 200ms (冷启动)
内存占用: ~30MB (基础)
工具注册: 16 个工具，< 10ms
```

## 6. 可扩展性

### 6.1 新增工具提供者

```python
class CustomToolProvider(BaseToolProvider):
    @property
    def provider_name(self) -> str:
        return "custom"

    def get_tools(self) -> List[ToolDefinition]:
        return [
            ToolDefinition(
                name="custom_action",
                description="自定义操作",
                inputSchema={
                    "type": "object",
                    "properties": {
                        "param1": {"type": "string"}
                    }
                }
            )
        ]

    async def call_tool(self, tool_name: str, arguments: dict) -> Any:
        if tool_name == "custom_action":
            return await self._do_custom(arguments["param1"])
        raise UnknownToolError(tool_name)

# 注册
registry = ToolRegistry()
registry.register_provider(CustomToolProvider())
```

### 6.2 新增平台适配

```python
# 在 api.py 中添加新的平台适配器
class NewPlatformHandler:
    def __init__(self, registry: ToolRegistry):
        self.registry = registry

    async def handle_request(self, platform_request: dict) -> dict:
        # 转换为内部工具调用
        result = await self.registry.call_tool(
            platform_request["tool"],
            platform_request["args"]
        )
        # 转换为平台格式
        return {"output": result}
```

## 8. 关键设计决策

### 8.1 为什么自研 Agent 而不是用 LangGraph

- **场景匹配**: 本项目的 Agent 是简单的工具调用管线，不需要 LangGraph 的图状态机
- **零依赖**: 不引入 LangChain 生态，降低部署复杂度
- **渐进式**: 预留接口，未来可接入 LangGraph

### 8.2 为什么自研缓存而不是用 LangChain Memory

- **缓存对象不同**: LangChain Memory 缓存对话，我们缓存工具调用结果
- **内容感知**: 基于 SHA256 的内容去重，而非简单 LRU
- **更轻量**: 纯 Python 标准库实现

### 8.3 为什么选择 JSON-RPC 2.0

- MCP 协议标准：Model Context Protocol 基于 JSON-RPC 2.0
- 简单可靠：请求/响应模型，易于调试
- 广泛支持：HTTP、SSE、STDIO 均支持

### 8.4 为什么用 aiohttp 而不是 FastAPI

- aiohttp 更底层，对 MCP 协议的 Streamable HTTP 和 SSE 支持更灵活
- 更小的依赖体积
- 原生支持 WebSocket（未来扩展）
