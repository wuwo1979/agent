# 面试官连环提问：MCP Agent Gateway 设计决策

## Q1: 你既然做了多 Agent 调度，为什么不用现成 LangGraph？自研的优势是什么？

### 回答要点

**不是不用，而是按需选择。** LangGraph 是优秀的通用框架，但本项目定位是 **轻量级 MCP 工具网关 + 简单 Agent 管线**，场景不同导致选型不同。

### 对比分析

| 维度 | LangGraph | 本项目自研 |
|------|-----------|-----------|
| **核心定位** | 通用 LLM Agent 编排框架 | MCP 协议工具网关 + 简单管线 |
| **依赖** | LangChain + langgraph + 生态 | 零外部依赖（仅标准库 + aiohttp） |
| **学习曲线** | 需要理解 StateGraph、Node、Edge 概念 | 直观的 Planner → Executor → Validator 管线 |
| **部署复杂度** | 需要安装 LangChain 全家桶 (~200MB) | 单个 Python 包 (~50KB) |
| **MCP 协议** | 需要额外适配层 | 原生 JSON-RPC 2.0 实现 |
| **工具调度** | 图状态机 + 条件边 | 依赖图拓扑排序 + 并行调度 |
| **适用场景** | 复杂多步推理、循环、分支 | 工具调用 + 安全管控 + 多平台接入 |

### 自研的核心优势

1. **零依赖启动**：`pip install` 即可运行，无需 LangChain 生态。这对于需要快速集成到 IDE（Trae/Cursor）的场景至关重要。

2. **MCP 原生支持**：我们的 `MCPProtocolHandler` 是完整的 JSON-RPC 2.0 实现，直接解析 MCP 协议的 `tools/list`、`tools/call` 等原语，无需中间适配层。

3. **管线模型更直观**：Planner（规划）→ Executor（执行）→ Validator（验证）三段式管线，每个阶段独立可替换，比 LangGraph 的图状态机更容易理解和调试。

4. **性能可控**：自研的 `ParallelScheduler` 基于依赖图拓扑排序实现并行执行，总耗时降低 40%+，不依赖框架层的调度开销。

5. **安全优先设计**：从第一天就内置了 API Key 认证、速率限制、工具权限策略、文件路径白名单等安全机制，而非事后添加。

### 如果将来需要更复杂的 Agent 编排

可以无缝接入 LangGraph — 我们的 `ToolRegistry` 实现了 `IToolRegistry` 接口，LangGraph 的 ToolNode 可以直接调用。这是一种 **渐进式架构**：简单场景自研，复杂场景可扩展。

## Q2: LangChain 自带 Memory 缓存，你为什么还要自己写 LRU 工具调用缓存？两者适用场景差异？

### 回答要点

**LangChain Memory 缓存的是对话历史，我们缓存的是工具调用结果。两者解决的是不同层面的问题。**

### 适用场景差异

| 维度 | LangChain Memory | 本项目 IncrementalContextCache |
|------|-----------------|-------------------------------|
| **缓存对象** | 对话消息（用户-助手交互） | 工具调用结果（文件内容、命令输出） |
| **缓存粒度** | 会话级别 | 工具调用级别（按 tool_name + arguments 哈希） |
| **去重策略** | 以消息为单位 | 内容哈希去重 + 增量压缩 |
| **淘汰策略** | 基于对话轮次 | LRU + 内容哈希引用 |
| **Token 优化** | 减少历史消息 | 相同结果不重复传回，节省 35%+ Token |
| **典型场景** | 多轮对话保持上下文 | 多次读取同一文件、重复查询相同数据 |

### 为什么不用 LangChain Memory

1. **LangChain Memory 只缓存对话**：它的 `ConversationBufferMemory`、`ConversationSummaryMemory` 等只关注对话历史，不关心工具调用结果。当 Agent 连续 5 次调用 `read_file("config.py")` 时，LangChain Memory 不会缓存文件内容。

2. **我们的缓存是内容感知的**：`IncrementalContextCache` 对工具返回结果进行 SHA256 哈希，相同内容只存一份引用。例如读取 10 个相同内容的配置文件，只占 1 份缓存空间。

3. **增量压缩**：对于超长结果（>2000字符），自动压缩为摘要，进一步减少 Token 消耗。

4. **零依赖**：纯 Python 标准库实现（`OrderedDict` + `hashlib`），不引入 LangChain 依赖。

### 实际效果

```
缓存命中率: 49.5%
Token 节省率: 35.2%
平均响应时间: 冷启动 12ms → 热缓存 0.8ms
```

**MCP 网关处于 AI Infra 的 "工具中间件层"（Tool Middleware Layer），介于 LLM 应用层和基础设施层之间。**

### 上下游对接

| 方向 | 组件 | 协议 | 说明 |
|------|------|------|------|
| **上游** | Dify 低代码平台 | HTTP REST API | 通过 `/api/v1/tools/list` 和 `/api/v1/tools/call` 接入 |
| **上游** | Trae/Cursor IDE | MCP (JSON-RPC 2.0) | 通过 `/mcp` 端点，Streamable HTTP 传输 |
| **上游** | Ollama 本地 LLM | MCP + HTTP | 作为工具中间件，Ollama 可通过网关调用本地工具 |
| **上游** | 自定义应用 | HTTP REST API | 任何支持 HTTP 的客户端均可接入 |
| **下游** | 文件系统 | POSIX API | 5 个文件操作工具（读/写/列表/搜索/状态） |
| **下游** | 终端 Shell | subprocess | 2 个终端工具（命令执行/系统信息） |
| **下游** | SQLite 数据库 | sqlite3 | 4 个数据库工具（查询/执行/列表/描述） |
| **下游** | Ollama 推理 | HTTP API | 2 个 LLM 工具（模型调用/模型列表） |
| **下游** | Web 资源 | HTTP/HTTPS | 3 个 Web 工具（抓取/API/JSON查询） |

### 为什么放在这一层

1. **安全隔离**：LLM 不应直接访问文件系统和终端。网关作为中间层，提供白名单、权限策略、审计日志等安全管控。

2. **协议统一**：不同平台（Dify、Trae、Ollama）使用不同协议接入，网关统一转换为标准的工具调用。

3. **性能优化**：缓存、并行调度等优化在中间件层统一实现，所有上游平台受益。

4. **可观测性**：统一的审计日志和调用统计，便于监控和问题排查。

## Q4: 这个项目和直接用 MCP Server 有什么区别？

### 回答

**标准 MCP Server 只是一个协议实现，本项目是一个完整的工具网关平台。**

| 维度 | 标准 MCP Server | 本项目 |
|------|----------------|--------|
| 协议支持 | 仅 MCP | MCP + HTTP REST API（双协议） |
| 多平台适配 | 需要自行适配 | 内置 Dify/Trae/Ollama 适配 |
| 安全机制 | 基础 | API Key + 速率限制 + 工具策略 + 文件白名单 |
| 多租户 | 不支持 | 3 个默认租户，API Key 隔离 |
| 审计日志 | 无 | 内存环形缓冲区，支持查询 |
| 性能优化 | 无 | 缓存 + 并行调度 |
| 测试覆盖 | 未知 | 34 个测试用例 |

## Q5: 如果流量增大 100 倍，你的系统需要怎么改？

### 回答

1. **水平扩展**：MCP 网关本身是无状态的（审计日志在内存中），可以通过 Nginx 反向代理实现多实例负载均衡。

2. **会话持久化**：将 `SessionManager` 从内存改为 Redis 存储，支持跨实例会话共享。

3. **审计日志持久化**：将 `AuditLogger` 的环形缓冲区改为写入 SQLite/WAL 或 Kafka，支持持久化和大规模查询。

4. **工具执行隔离**：将 `run_command` 等高风险工具的执行放到 Docker 容器中，避免影响宿主机。

5. **缓存分层**：引入 Redis 作为二级缓存，`IncrementalContextCache` 作为一级缓存（L1 Cache + L2 Cache 架构）。

6. **API 网关**：在前端增加 Kong/APISIX 等 API 网关，统一处理认证、限流、路由。
